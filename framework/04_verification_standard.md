## Required Data Package

Every R³ application must produce the following before claiming a fixed point:

### 1. Dissolution Log

Complete record of every layer identified and every layer tested.

```
For each layer:
─────────────────────────────
Layer identifier
Layer description
Layer type (IC / PA / AR / PP / HA)
Reason for identification
Test performed
Result of test
State change observed (yes/no)
Decision: remove or keep
Justification for decision
```

**The dissolution log must be complete.**  
Selective logging is contamination (C3).

---

### 2. Fixed Point Specification

Complete technical description of the bedrock state.

```
Must include:
─────────────────────────────
All structural parameters
All measurable properties
All predicted observable signatures
Stability confirmation
Iteration depth required
```

Parameters must be specific and quantitative.  
"High performance" is not a fixed point specification.  
"Tc = 295 ± 2 K" is a fixed point specification.

---

### 3. Bridge Document

Explicit connection between fixed point and established knowledge.

```
Must demonstrate:
─────────────────────────────
The relevant established theory
How the fixed point maps to it
The specific mechanism
The derivation or calculation
Where theory is sufficient
Where theory must be extended
What gaps remain
```

The bridge document must acknowledge its own gaps honestly.  
A bridge document that claims no gaps is contaminated.

---

### 4. Verification Protocol

Step-by-step instructions any qualified party can follow.

```
Must include:
─────────────────────────────
Complete materials list with specifications
Complete equipment list with specifications
Step-by-step procedure
Acceptance criteria for each test
Expected results if fixed point exists
Expected results if fixed point does not exist
Troubleshooting guidance
Safety requirements
```

If a verification protocol cannot be written, the fixed point is not yet specified.  
A fixed point that cannot be tested is not a fixed point. It is speculation.

---

### 5. Verification Results

Record of every attempt to verify the fixed point.

```
For each attempt:
─────────────────────────────
Date
Operator name and credentials
Institution
Equipment used (with calibration status)
Procedure followed (note any deviations)
Raw data (complete, unprocessed)
Analysis method
Result: CONFIRM / DENY / INCONCLUSIVE
Interpretation
Data storage location (permanent, accessible)
```

## Reporting Requirements

### All Results Must Be Reported

There is no such thing as a negative result that should be suppressed.

A negative result is:
- Evidence that the fixed point specification needs refinement
- Evidence that the synthesis or implementation protocol needs improvement
- Evidence that the bridge document needs revision
- Occasionally: evidence that the fixed point does not exist

All of these outcomes advance knowledge.  
All of these outcomes must be reported.

### Selective Reporting Is Contamination

Reporting only positive results violates C3 (no confusion over uncertainty) and C6 (reflection reveals hidden structure).

If only positive results are reported, the reflection is incomplete. Hidden structure — including disconfirming evidence — remains hidden.

### Raw Data Always

Interpreted results alone are insufficient.  
Raw data must be included with every report.  
Anyone must be able to reanalyze the raw data independently.

### Interpretation Clearly Labeled

Distinguish:
- What was measured (fact)
- What the measurement means (interpretation)
- What the interpretation implies (inference)

Label each clearly. Do not present interpretation as measurement.

## The Verification Ladder

```
Level 1: Self-verification
─────────────────────────────
Author verifies own work.
Necessary but insufficient.
Cannot eliminate blind spots.


Level 2: Independent replication
─────────────────────────────
Different operator.
Same institution.
Same protocol.
Stronger than Level 1.


Level 3: Independent replication
─────────────────────────────
Different operator.
Different institution.
Same protocol.
Standard scientific confirmation.


Level 4: Independent replication
─────────────────────────────
Different operator.
Different institution.
Different protocol.
Same result.
Strong confirmation.


Level 5: Predictive confirmation
─────────────────────────────
Fixed point predicts new observable
not known at time of specification.
New observable confirmed.
Strongest possible confirmation.
```

R³ applications should aim for Level 3 minimum before making public claims.

## Submitting Results

All verification results should be submitted to:

1. **This repository** — `verification/submissions/`
2. **Open access preprint servers** — arXiv, ChemRxiv, bioRxiv, SSRN as appropriate
3. **Peer-reviewed journals** — encouraged but not required for submission here
4. **Directly to the author** — Michael@advanceerivs.com

**Format for submission:**

```
File naming: R3-VERIFY-[INDUSTRY]-[DATE]-[RESULT].zip

Contents:
─────────────────────────────
verification_report.md (structured per template above)
raw_data/ (all raw data files)
analysis/ (analysis scripts and outputs)
photos/ (photographs of setup and results)
README.txt (brief description)
```
